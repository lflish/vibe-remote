# 协议单一事实来源:Go → TS 代码生成

**日期**: 2026-07-20
**状态**: 设计待评审

## 问题

`vibe-remoted/internal/protocol/protocol.go` 与 `desktop/src/shared/protocol.ts` 是两份**手写**的协议定义,靠人工保持对齐(CLAUDE.md 明确写「必须手动保持对齐」)。改一个帧的字段容易漏改另一端,造成**运行时静默错位**——JSON 字段名或类型不匹配时,TS 端不报错,只是拿到 `undefined`,是最难排查的一类故障。

当前有 11 种帧类型,TS 侧完全是 Go struct 的镜像。

## 目标

`protocol.go` 成为协议的**唯一手写源**。`protocol.ts` 变成生成产物,顶部标注 `// Code generated by gen.go; DO NOT EDIT.`。改协议只动 Go 一处,`make proto` 重写 TS。CI / 提交前用 `make proto-check` 校验「生成后 git diff 干净」,漏同步直接红。

**非目标(YAGNI)**:不引入 zod 等运行时校验;不做中立 IDL;不生成 Go 端。只解决「TS 相对 Go 漂移」这一个痛点。

## 关键难点

TS 侧比 Go struct 多三类信息,Go 本身表达不了,生成器必须有确定来源:

| TS 独有 | 现状 | 来源方案 |
|---|---|---|
| `FrameType` const map | 手写 | 解析 Go 的 `const ( TypeAuth = "auth" … )` 块 |
| `ClientFrame` / `ServerFrame` union(方向) | 手写 | Go 侧新增结构化方向声明(见下) |
| `DataFrameC2S` / `DataFrameS2C`(同结构拆两份) | 手写重复 | data 声明为双向,生成单个 `DataFrame` interface,同时进两个 union |

## 方案

### 1. Go 侧新增方向声明(方向的单一来源)

在 `protocol.go` 追加一段**编译期可检查**的方向声明,取代 TS 侧手写的 union:

```go
// FrameDirections declares which frames flow which way. The generator reads
// this to emit the ClientFrame / ServerFrame TypeScript unions. Because these
// are real type literals, the compiler guarantees every referenced frame exists
// — direction info cannot silently drift. Keep in sync when adding a frame.
var _ = frameDirections{
	Client: []any{AuthFrame{}, AttachFrame{}, DataFrame{}, ResizeFrame{}, PingFrame{}},
	Server: []any{ReadyFrame{}, DataFrame{}, SessionsFrame{}, ExitFrame{}, ErrorFrame{}, NotifyFrame{}, PongFrame{}},
}

type frameDirections struct {
	Client []any
	Server []any
}
```

比注释魔法更可靠:引用不存在的 struct 会**编译失败**,方向信息无法悄悄漂移。

**副作用(正向)**:当前 Go 侧只有 `TypePing`/`TypePong` 常量,没有对应空 struct。此方案要求补齐 `PingFrame{}` / `PongFrame{}` 两个空 struct(仅含 `Type` 字段),让两端真正对称。

### 2. `DataFrame` 去重

Go 端已有单个 `DataFrame`。TS 端现在手写了 `DataFrameC2S` 和 `DataFrameS2C` 两个同构 interface。生成器让 `DataFrame` 因同时出现在 Client 和 Server 列表中而**只生成一次**,两个 union 都引用它。比现状更干净,消除人为重复。

> 影响:`client.ts` 里 `this.send<DataFrameC2S>(…)` 需改为 `this.send<DataFrame>(…)`,`DataFrameS2C` 的引用同理。属机械改名。

### 3. 生成器实现

- **文件**: `vibe-remoted/internal/protocol/gen.go`,首行 `//go:build ignore`(不进正常编译)。
- **触发**: `protocol.go` 顶部加 `//go:generate go run gen.go`。
- **解析方式**: 用标准库 `go/ast` + `go/parser` **静态解析** `protocol.go` 单文件。不用 reflection——静态解析能拿到 const 块、能读取字段行尾注释(如 `// empty = create new`)并原样带进 TS。
- **类型映射**: `string→string`、`int→number`、`uint16→number`、`bool→boolean`、`[]T→T[]`、结构体名→同名 interface。字段名取 json tag;带 `,omitempty` 的生成 TS 可选字段(`field?:`)。
- **输出**: 覆写 `desktop/src/shared/protocol.ts`,顶部 `// Code generated by gen.go; DO NOT EDIT.`。
- **顺序**: 生成内容按稳定顺序输出(源码声明顺序),保证同一输入产出字节一致的文件——`proto-check` 的 diff 才有意义。

### 4. `MachineConfig` 挪出

`MachineConfig` 是纯客户端配置类型,不属于协议。它当前在 `protocol.ts` 末尾。

- 新建 `desktop/src/shared/machine-config.ts`,内容为 `MachineConfig` interface。
- 生成后的 `protocol.ts` 变成 **100% 纯生成、无手写区**。
- 更新 8 个引用文件的 import(`type { MachineConfig }` 从 `../shared/protocol` 改为 `../shared/machine-config`):
  `client.ts`、`rest.ts`、`dirpicker.ts`、`machines.ts`、`index.ts`(renderer)、`preload.ts`、`index.ts`(main)。
  其中 `rest.ts` 和 renderer `index.ts` 同时 import 了 `MachineConfig` 和 `SessionInfo`,需拆成两条 import(`SessionInfo` 仍来自 `protocol`)。

### 5. 校验闭环

Makefile 新增:

```makefile
# Regenerate desktop protocol.ts from protocol.go (single source of truth)
proto:
	cd vibe-remoted/internal/protocol && go generate

# Verify protocol.ts is in sync with protocol.go (CI / pre-commit)
proto-check:
	@cd vibe-remoted/internal/protocol && go run gen.go --check
```

`gen.go` 支持 `--check`:生成到内存,与磁盘上的 `protocol.ts` 比对,不一致则打印 diff 提示并**非零退出**。

### 6. 文档更新

- `CLAUDE.md`:把「Go 端…与 TS 端…**必须手动保持对齐**(无代码生成)」改为「protocol.ts 由 protocol.go 生成,改协议只改 protocol.go 后跑 `make proto`;`make proto-check` 守护同步」。
- `docs/protocol.md`:补一句「TS 定义为生成产物,勿手改」。

## 数据流

```
protocol.go (唯一手写源)
  ├─ struct 定义 ──┐
  ├─ const 块 ─────┼─→ gen.go (go/ast 静态解析) ─→ protocol.ts (生成, DO NOT EDIT)
  └─ frameDirections ┘                              machine-config.ts (手写, 挪出)

make proto        → 重新生成
make proto-check  → 生成到内存 diff 磁盘, 不一致非零退出 (CI 守门)
```

## 测试策略

- **生成器正确性**: 跑 `make proto` 后,现有 `cd desktop && npm run typecheck` 必须通过(TS 编译 = 生成结果结构正确的最强证明,因为所有引用点都会被类型检查)。
- **同步守护**: `make proto-check` 在干净状态下必须零退出;手动改坏 `protocol.ts` 后必须非零退出(冒烟验证一次)。
- **回归**: `cd vibe-remoted && go test ./...` 与 `go vet ./...` 保持通过(新增的 `frameDirections` 声明和空 struct 不能破坏现有测试)。
- **端到端**: 生成后 `make dev-local` 自连冒烟,确认帧收发正常(纯字节透传链路不受影响)。

## 风险与权衡

- **改动面**: 挪 `MachineConfig` 触及 8 个文件 import,比预想多,但全是机械改动,IDE 可批量完成,不影响运行时。
- **go/ast 解析脆弱性**: 若日后有人在 `protocol.go` 用了生成器不认识的类型(如 map、嵌套匿名 struct),生成器应**显式报错而非静默跳过**——宁可生成失败也不产出错误 TS。当前 11 种帧只用到 string/int/uint16/slice,覆盖足够;新类型由生成器 fail-fast 兜底。
- **单向生成的局限**: 只防 TS 漂移,不防有人手改 Go 后忘跑生成。`proto-check` 进 CI 后此风险归零(忘跑 = CI 红)。
